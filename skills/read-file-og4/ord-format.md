# Oribos Reader .ord Format

`.ord` files are plain XML files (not zipped) generated by **Oribos Reader**, the chip-reading companion app. They record raw chip readouts from SportIdent stations (SI cards) during an orienteering event, along with reader configuration and manual modifications made at the reading station.

## Root Element

```xml
<?xml version="1.0" encoding="utf-8" standalone="yes"?>
<Program type="ORIBOS READER" version="300">
  ...configuration...
  <mods>...</mods>
  <SiCard ...>...</SiCard>
  <SiCard ...>...</SiCard>
  ...
</Program>
```

---

## Configuration Fields

Direct children of `<Program>`, before `<mods>` and `<SiCard>` entries.

| Field | Example | Description |
|---|---|---|
| `Sistema` | `SportIdent` | Timing system type |
| `Port` | `COM6` | Serial port for the reader station |
| `BaudRate` | `38400` | Serial baud rate |
| `NomeFileDati` | *(empty)* | Data file name (optional) |
| `NomeFileOribos` | *(empty)* | Link to the .og4 file (optional) |
| `UsaFileOribos` | `True` | Whether to send data to the Oribos competition file |
| `UsaTempoOribos` | `True` | Whether to use the zero-time from the Oribos file |
| `T3` | `10:00:00` | Reference time (T3 = zero-time/mass start time) |
| `UsaStampante` | `True` | Whether to use a receipt printer |
| `Stampante` | `EPSON TM-T20II Receipt5` | Printer name |
| `TipoPagina` | `0` | Page/print layout type |
| `CalcolaTempi` | `False` | Whether to compute split times locally |
| `UseOldProtocol` | `False` | Use legacy SI protocol |
| `ProgMod` | `3` | Progressive modification counter |
| `Giorno` | `1` | Day number (for MultiDays events) |
| `UsaWebServer` | `True` | Whether to publish to the web server |
| `Backup1` | *(empty)* | Backup path 1 |
| `Backup2` | *(empty)* | Backup path 2 |
| `AvvisaErroreBackup` | `False` | Alert on backup failure |

## Modifications — `<mods>`

The `<mods>` section logs manual changes made by the operator at the reading station (e.g., reassigning a chip number or changing a category). Each `<mod>` entry records one change.

```xml
<mods>
  <mod>
    <Tipo>SiCard</Tipo>      <!-- What was changed: SiCard | Categoria -->
    <Prog>1</Prog>           <!-- Progressive number of this modification -->
    <Id>8275</Id>             <!-- ID of the athlete record in the .og4 file -->
    <OldValue>8636055</OldValue>  <!-- Previous value -->
    <NewValue>8140589</NewValue>  <!-- New value -->
  </mod>
</mods>
```

**`Tipo` values:**
- `SiCard` — chip number was changed (e.g., athlete used a different chip than registered)
- `Categoria` — category was changed at readout time

## Chip Readouts — `<SiCard>`

Each `<SiCard>` element represents one chip readout. A chip may appear multiple times if re-read (e.g., duplicate reads, or a re-read after disabling a previous one).

### Attributes

```xml
<SiCard number="8494383" type="5" volt="2.89" batt="19-12-10" disable="true">
```

| Attribute | Required | Description |
|---|---|---|
| `number` | Yes | SI card number |
| `type` | Yes | SI card type (`5` = SI-5/6/8/9/10/11, etc.) |
| `volt` | No | Battery voltage of the SI card (only on cards that report it) |
| `batt` | No | Battery date of the SI card (format `YY-MM-DD`) |
| `disable` | No | `"true"` if this readout was invalidated (e.g., replaced by a later read) |

### Child Elements

```xml
<SiCard number="2099329" type="5">
  <Start>09.31.51,0</Start>
  <Finish>10.11.37,0</Finish>
  <Check>09.28.58</Check>
  <Read>01/25/2026 10:19:32</Read>
  <Cat>Rosso M</Cat>
  <St>CLA</St>
  <Final>00.39.46</Final>
  <Pts>0</Pts>
  <ms>true</ms>
  <SplitTime prog="0">
    <Code>36</Code>
    <Time>09.33.37</Time>
  </SplitTime>
  ...
</SiCard>
```

| Field | Format | Description |
|---|---|---|
| `Start` | `HH.MM.SS,d` | Start punch time. `00.00.00,0` = no start punch (used zero-time). The `,d` suffix is a sub-second digit (tenths) |
| `Finish` | `HH.MM.SS,d` | Finish punch time. Same format as Start |
| `Check` | `HH.MM.SS` | Check station punch time (when athlete cleared the card) |
| `Read` | `MM/DD/YYYY HH:MM:SS` | Timestamp when the card was read at the download station |
| `Cat` | text | Category name (e.g., `M14`, `Rosso M`). Empty if unknown at read time |
| `St` | text | Status code: `CLA` = classified. Empty if not yet classified |
| `Final` | `HH.MM.SS` | Final time (computed). `00.00.00` if not computed |
| `Pts` | integer | Points (for Score-O). `0` for normal races |
| `ms` | `true` | Present when the card supports sub-second (millisecond) timing. Absent otherwise |

### Split Times

Each `<SplitTime>` records one control punch, in the order stored on the chip.

```xml
<SplitTime prog="0">
  <Code>36</Code>
  <Time>09.33.37</Time>
</SplitTime>
```

| Field | Description |
|---|---|
| `prog` | 0-based index of this punch in the chip's memory |
| `Code` | Control station code number. `100` is commonly used for a beacon/spectator control near the finish |
| `Time` | Absolute punch time (`HH.MM.SS`) |

**Note:** The split times are raw chip data — they follow the order the athlete punched, which may not match the expected course order. Validation against the course (from the .og4 file) is done by Oribos, not by the reader.

## Key Differences from .og4

| Aspect | .og4 (Atleti.xml) | .ord |
|---|---|---|
| Format | ZIP archive with multiple XML files | Single plain XML file |
| Times | Encoded as integer centiseconds | Human-readable `HH.MM.SS` format |
| Athlete data | Full (name, club, category, bib, etc.) | Only chip number + what's resolved at read time |
| Split times | Validated against course order | Raw chip order (as punched) |
| Purpose | Competition management & results | Raw chip download log |
| Duplicates | One record per athlete | May have multiple reads of the same chip |
